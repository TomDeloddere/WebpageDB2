Authentication Setup Exercises

Practice configuring authentication mechanisms for DB2 on z/OS.

Exercise 1: Configure RACF User for DB2 Access

Create a RACF user profile for a new DB2 user (DBUSER1) and permit access to a DB2 subsystem (DB2X).

Requirements:

  • Create a RACF user with a password
  • Assign to a default group (DBGROUP)
  • Permit READ access to the DB2 subsystem resource
  • Provide RACF commands
  • Explain the importance of ESM integration
// RACF Commands
ADDUSER DBUSER1 DFLTGRP(DBGROUP) PASSWORD(TEMPPASS) NAME('DB2 User One')

PERMIT DSN.DB2X.SYSUSER CLASS(DSNR) ID(DBUSER1) ACCESS(READ)

Explanation: The ADDUSER command creates a RACF user profile for DBUSER1 with a temporary password and assigns it to the DBGROUP group. The PERMIT command grants READ access to the DB2X subsystem resource in the DSNR class, allowing DBUSER1 to connect to DB2. ESM (RACF) integration centralizes authentication, ensuring consistent security policies across z/OS and DB2, leveraging RACF's robust user management and auditing capabilities.

Privilege Management Exercises

Practice granting and managing DB2 privileges and authorities.

Exercise 2: Grant Database-Level Authority

Grant DBADM authority on a database (HRDB) to a user (DBA01) without data access privileges.

Requirements:

  • Use GRANT statement
  • Ensure no inherent data access
  • Provide SQL statement
  • Explain the implications of DBADM
GRANT DBADM WITHOUT DATAACCESS ON DATABASE HRDB TO USER DBA01;

Explanation: The GRANT statement assigns DBADM authority to DBA01 for the HRDB database, with WITHOUT DATAACCESS ensuring no automatic SELECT, INSERT, UPDATE, or DELETE privileges on tables. DBADM allows full administrative control over the database (e.g., creating tables, running utilities, granting privileges), but explicit data access grants are needed for data operations, supporting separation of duties.

Role-Based Access Control Exercises

Practice implementing RBAC using roles.

Exercise 3: Create and Assign a Role for Analysts

Create a role for data analysts (ANALYST_ROLE) and grant it specific privileges on a table (SALES_DATA).

Requirements:

  • Create the role
  • Grant SELECT and EXECUTE privileges
  • Assign the role to a user (ANALYST01)
  • Provide SQL statements
  • Explain the benefits of RBAC
CREATE ROLE ANALYST_ROLE;

GRANT SELECT ON TABLE MYSCHEMA.SALES_DATA TO ROLE ANALYST_ROLE;
GRANT EXECUTE ON PROCEDURE MYSCHEMA.ANALYZE_SALES TO ROLE ANALYST_ROLE;

GRANT ROLE ANALYST_ROLE TO USER ANALYST01;

Explanation: The role ANALYST_ROLE is created and granted SELECT on SALES_DATA and EXECUTE on a stored procedure, enabling data analysis tasks. Assigning the role to ANALYST01 grants these privileges indirectly. RBAC simplifies privilege management by grouping permissions by job function, enhances consistency, and reduces administrative overhead when users change roles or leave.

Discretionary Access Control Exercises

Practice managing permissions with DAC.

Exercise 4: Grant and Revoke Privileges with Grant Option

Grant SELECT privilege on a table (CUSTOMER_INFO) to a user (MGR01) with the ability to grant it to others, then revoke only the grant option.

Requirements:

  • Use GRANT and REVOKE statements
  • Ensure MGR01 retains SELECT privilege
  • Provide SQL statements
  • Explain the risks of WITH GRANT OPTION
GRANT SELECT ON TABLE MYSCHEMA.CUSTOMER_INFO TO USER MGR01 WITH GRANT OPTION;

REVOKE GRANT OPTION FOR SELECT ON TABLE MYSCHEMA.CUSTOMER_INFO FROM USER MGR01;

Explanation: The GRANT statement gives MGR01 SELECT privilege and the ability to grant it to others. The REVOKE statement removes only the grant option, preserving MGR01's SELECT privilege. WITH GRANT OPTION risks privilege proliferation, as grantees can extend access to others, potentially leading to unintended access. Careful tracking via SYSIBM.SYSTABAUTH is needed to manage dependencies.

Label-Based Access Control Exercises

Practice implementing LBAC for multi-level security.

Exercise 5: Implement LBAC for Classified Data

Create an LBAC policy for a table (PROJECT_FILES) to enforce access based on security levels and compartments.

Requirements:

  • Create a security policy with levels and compartments
  • Define security labels
  • Apply to the table
  • Provide SQL statements
  • Explain LBAC enforcement
CREATE SECURITY LABEL COMPONENT LEVEL ('TOP_SECRET', 'SECRET', 'UNCLASSIFIED');
CREATE SECURITY LABEL COMPONENT COMPARTMENT ('ALPHA', 'BETA');

CREATE SECURITY POLICY CLASSIFIED_POLICY COMPONENTS LEVEL, COMPARTMENT;

CREATE SECURITY LABEL CLASSIFIED_POLICY.TS_ALPHA 
    COMPONENT LEVEL 'TOP_SECRET', 
    COMPONENT COMPARTMENT 'ALPHA';
CREATE SECURITY LABEL CLASSIFIED_POLICY.S_BETA 
    COMPONENT LEVEL 'SECRET', 
    COMPONENT COMPARTMENT 'BETA';

ALTER TABLE MYSCHEMA.PROJECT_FILES 
    ADD COLUMN DATA_LABEL SECPOLICYNAME CLASSIFIED_POLICY;

Explanation: The security policy CLASSIFIED_POLICY uses LEVEL and COMPARTMENT components. Labels TS_ALPHA and S_BETA define specific access criteria. Adding a DATA_LABEL column to PROJECT_FILES enables LBAC. DB2 enforces access by comparing user labels (granted via SECADM) to row labels, ensuring users only access rows where their label dominates (e.g., TOP_SECRET users access SECRET rows, but not vice versa).

Row and Column Access Control Exercises

Practice configuring RCAC for fine-grained access.

Exercise 6: Create Row Permission and Column Mask

Implement RCAC on a table (EMPLOYEE_DATA) to restrict row access to managers' departments and mask SSN for non-HR users.

Requirements:

  • Create a row permission for department-based access
  • Create a column mask for SSN
  • Provide SQL statements
  • Explain RCAC benefits
CREATE PERMISSION MGR_DEPT_ACCESS ON MYSCHEMA.EMPLOYEE_DATA
    FOR ROWS WHERE DEPARTMENT_ID = 
        (SELECT DEPT_ID FROM MYSCHEMA.MANAGERS WHERE MGR_USERID = SESSION_USER)
    ENFORCED FOR ALL ACCESS ENABLE;

CREATE MASK SSN_MASK ON MYSCHEMA.EMPLOYEE_DATA
    FOR COLUMN SSN RETURN
        CASE WHEN VERIFY_ROLE_FOR_USER(SESSION_USER, 'HR_ROLE') = 1
             THEN SSN
             ELSE 'XXX-XX-' || SUBSTR(SSN, 8, 4)
        END
    ENABLE;

Explanation: The row permission restricts access to rows where the user's department matches their manager record. The column mask hides SSN for non-HR users, showing only the last four digits. RCAC provides fine-grained, transparent access control, reducing application changes, enhancing security, and supporting compliance by limiting sensitive data exposure.

Trusted Contexts Exercises

Practice setting up trusted contexts for secure connections.

Exercise 7: Create a Trusted Context for Application Server

Define a trusted context for an application server (APPSRV1) to assign a role (APP_USER_ROLE) to connections.

Requirements:

  • Specify server AUTHID and IP address
  • Assign a default role
  • Provide SQL statement
  • Explain trusted context use cases
CREATE TRUSTED CONTEXT APP_SERVER_CTX
    BASED UPON CONNECTION USING SYSTEM AUTHID APPSRV1
    ATTRIBUTES (ADDRESS '192.168.1.100')
    DEFAULT ROLE APP_USER_ROLE
    ENABLE;

Explanation: The trusted context assigns APP_USER_ROLE to connections from APPSRV1 at the specified IP, ensuring only trusted connections gain the role's privileges. Trusted contexts are ideal for middleware servers, enabling secure user switching and connection pooling without repeated authentication, enhancing security and performance in multi-tier applications.

Auditing Exercises

Practice configuring auditing for security events.

Exercise 8: Create an Audit Policy for Sensitive Data

Create an audit policy to track access to a sensitive table (CUSTOMER_CREDIT_INFO).

Requirements:

  • Audit all access types
  • Provide SQL statements
  • Explain auditing benefits
CREATE AUDIT POLICY CREDIT_AUDIT
    CATEGORIES ALL STATUS BOTH ERROR TYPE NORMAL;

ALTER TABLE MYSCHEMA.CUSTOMER_CREDIT_INFO 
    AUDIT POLICY CREDIT_AUDIT;

Explanation: The audit policy CREDIT_AUDIT logs all access (SELECT, INSERT, UPDATE, DELETE) to CUSTOMER_CREDIT_INFO, capturing both successful and failed attempts. Auditing provides visibility into data access, supports compliance (e.g., GDPR, PCI DSS), and enables forensic analysis of security incidents, ensuring accountability.

Encryption Exercises

Practice enabling encryption for data protection.

Exercise 9: Enable Tablespace Encryption

Configure a tablespace (SENSITIVE_TS) to use native DB2 encryption.

Requirements:

  • Use a key label in ICSF
  • Provide SQL statement
  • Explain encryption considerations
ALTER TABLESPACE MYDB.SENSITIVE_TS 
    USING STOGROUP SGSECURE
    ENCRYPTION KEY LABEL 'MY_ENCR_KEY_LABEL';

Explanation: The ALTER TABLESPACE statement enables encryption for SENSITIVE_TS using a key label managed in ICSF. Native encryption protects data at rest transparently, but requires secure key management to prevent data loss and coordination with z/OS cryptographic services. Performance impacts and backup procedures must also be considered.

Compliance Planning Exercises

Practice developing plans to meet regulatory requirements.

Exercise 10: Develop a GDPR Compliance Plan

Create a plan to ensure GDPR compliance for a DB2 table (PERSONAL_DATA) containing personal information.

Requirements:

  • Outline steps for access control, auditing, and encryption
  • Provide a text-based procedure
  • Explain GDPR relevance
GDPR COMPLIANCE PLAN FOR PERSONAL_DATA

1. Implement Access Control
   - Action: Create roles (e.g., DATA_PROCESSOR_ROLE) and grant minimal privileges (SELECT, UPDATE) on PERSONAL_DATA.
   - SQL: 
     CREATE ROLE DATA_PROCESSOR_ROLE;
     GRANT SELECT, UPDATE(NAME, EMAIL) ON MYSCHEMA.PERSONAL_DATA TO ROLE DATA_PROCESSOR_ROLE;
     GRANT ROLE DATA_PROCESSOR_ROLE TO USER PROC01;
   - Rationale: Enforces least privilege, limiting access to authorized users.

2. Configure RCAC for Data Minimization
   - Action: Use row permissions to restrict access to relevant data and column masks to hide sensitive fields (e.g., SSN).
   - SQL: 
     CREATE PERMISSION DATA_ACCESS ON MYSCHEMA.PERSONAL_DATA
       FOR ROWS WHERE REGION = (SELECT USER_REGION FROM USER_PROFILES WHERE USERID = SESSION_USER)
       ENFORCED FOR ALL ACCESS ENABLE;
     CREATE MASK SSN_MASK ON MYSCHEMA.PERSONAL_DATA
       FOR COLUMN SSN RETURN
         CASE WHEN VERIFY_ROLE_FOR_USER(SESSION_USER, 'HR_ROLE') = 1 THEN SSN ELSE 'XXX-XX-XXXX' END
       ENABLE;
   - Rationale: Minimizes data exposure, aligning with GDPR's data minimization principle.

3. Enable Auditing
   - Action: Create an audit policy to log all access to PERSONAL_DATA.
   - SQL: 
     CREATE AUDIT POLICY GDPR_AUDIT
       CATEGORIES ALL STATUS BOTH ERROR TYPE NORMAL;
     ALTER TABLE MYSCHEMA.PERSONAL_DATA AUDIT POLICY GDPR_AUDIT;
   - Rationale: Provides evidence of data access for GDPR accountability and breach investigations.

4. Encrypt Data
   - Action: Enable native encryption for the tablespace containing PERSONAL_DATA.
   - SQL: 
     ALTER TABLESPACE MYDB.PERSONAL_TS ENCRYPTION KEY LABEL 'GDPR_KEY';
   - Rationale: Protects personal data at rest, meeting GDPR's security requirements.

5. Document Policies
   - Action: Maintain documentation of security configurations, access policies, and audit procedures.
   - Tool: Internal documentation system.
   - Rationale: GDPR requires proof of compliance through documented controls.

GDPR RELEVANCE:
GDPR mandates strong protection for personal data, including access control, data minimization, encryption, and auditability. This plan uses DB2 features to enforce these requirements, ensuring compliance and reducing the risk of fines or reputational damage.

Explanation: The plan leverages RBAC, RCAC, auditing, and encryption to meet GDPR requirements. Roles and RCAC ensure only authorized users access relevant data, auditing tracks all actions, and encryption secures data at rest. Documentation supports compliance audits, aligning with GDPR's emphasis on accountability and data protection.
